## 🧩 Topic 1 — Syntax

**Cover:** Variables, data types, operators, if/else, loops (`for`, `while`), functions, basic I/O.

**How to approach:**
- Type out every example — don't copy-paste.
- Write small programs yourself: sum of N numbers, check even/odd, print a pattern.
- If something doesn't work, read the error, don't just Google it immediately.

> **Remember:** You don't need to memorize syntax. Understand the logic — it'll stick through practice.

---

## ⏱️ Topic 2 — Time & Space Complexity

**Cover:** Big O notation — O(1), O(n), O(n²), O(log n). How to read the complexity of loops. Worst vs best case.

**How to approach:**
- Learn to *label* code you've already written with its complexity.
- Quick rule of thumb:

```
One loop over n      →  O(n)
Loop inside a loop   →  O(n²)
Input halved each step  →  O(log n)
No loop at all       →  O(1)
```

- For every problem you solve from here on, always ask: *what is my complexity, and can I do better?*

> **Remember:** Don't stress about formal proofs. Focus on pattern recognition for now.

---

## 📦 Topic 3 — Arrays

**Cover:** Indexing, traversal, insert/delete/search, reverse, prefix sums, two-pointer (intro).

**How to approach:**
- Draw the array on paper before writing code. Label your indices.
- Start with the classics: max/min element, reverse, check if sorted, rotate by K.
- Then level up: move zeros to end, find duplicates, subarray sum.

> **Remember:** Most array bugs are off-by-one errors. Always double-check `i < n` vs `i <= n`.

---

## 🔢 Topic 4 — Sorting

**Cover (in this order):** Bubble Sort → Selection Sort → Insertion Sort → Merge Sort → Quick Sort.

**How to approach:**
- Implement Bubble, Selection, Insertion **from scratch** — they're simple enough to do by logic alone.
- For Merge Sort, **trace through an example on paper first**, then code.
- Test every sort on: sorted input, reverse-sorted input, array with duplicates.

**Complexity at a glance:**

| Algorithm | Time (Avg) | Space |
|-----------|------------|-------|
| Bubble / Selection / Insertion | O(n²) | O(1) |
| Merge Sort | O(n log n) | O(n) |
| Quick Sort | O(n log n) | O(log n) |

> **Remember:** Don't skip the slow sorts. They teach you *how to think* about algorithms.

## 🧪 Daily Problem Set — How to Use It

This is the most important part of your week. One problem per day, every day.

**Your approach for each problem:**
1. Read the problem **twice**.
2. Think for **at least 15–20 minutes** before writing any code.
3. Code the brute force first — a slow working solution beats no solution.
4. Check: what's the time complexity? Can you optimize?
5. Test with edge cases: empty input, single element, all duplicates.

**If you're stuck:**
- Look at the hint/approach, not the full solution.
- After solving (even with help), **re-solve it from scratch the next day**. That's where real learning happens.

**Track your progress** — even a simple log helps:
```
Day | Problem | Solved? | Approach | Complexity | Notes
```
